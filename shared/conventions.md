# Конвенции vassal-litigator

## Расположение документов

**Правило:** Все выходные документы, предназначенные для Сюзерена (юриста), сохраняются в корневой папке дела (CASE ROOT). Папка `.vassal/` зарезервирована исключительно для внутренних агентских файлов и не должна быть доступна пользователю.

Все файлы и папки в корне дела начинаются с даты `ГГГГ-ММ-ДД`.

Примеры:
- Предварительный анализ: `{case-root}/2026-03-26 Предварительный анализ документов.md`
- Позиция по делу: `{case-root}/2026-03-26 Начальная позиция.md`
- Процессуальные документы: `{case-root}/2026-04-01 Истец - исковое заявление/`

Внутренние агентские файлы (которые агент читает, но не показывает Сюзерену):
- `.vassal/case.yaml` — карточка дела
- `.vassal/index.yaml` — реестр документов
- `.vassal/mirrors/` — md-зеркала документов
- `.vassal/analysis/{qualification.md, deadlines.md, ...}` — аналитические блоки (для переиспользования между скиллами)
- `.vassal/history.md` — история действий агента
- `.vassal/raw/` — оригиналы загруженных документов
- `.vassal/plans/` — временные планы файловых операций, согласованные Сюзереном; удаляются apply-фазой
- `.vassal/work/` — рабочие артефакты plan-фазы (распакованные архивы, временный OCR); удаляются apply-фазой

Для дел, начатых на `v0.5.x`, в `.gitignore` и на диске могут сохраняться legacy-папки предыдущего рантайма. `v0.6.0` их не использует и не обновляет.

## Именование файлов

Формат: `ГГГГ-ММ-ДД Отправитель Описание.расширение`

Правила:
- Дата — дата документа (не дата создания файла), формат строго `ГГГГ-ММ-ДД`
- Отправитель — организация или лицо, без кавычек: `ООО «Ромашка»` → `ООО Ромашка`
- Описание — тип документа и/или суть. Номер документа, если есть
- Расширение — никогда не менять, **кроме** скриншотов и картинок (см. ниже)
- Кодировка — UTF-8, кириллица, без транслита
- Если дата неизвестна и файл **нельзя привязать** к документу-узлу — см. §«Сироты без даты»
- Если отправитель неизвестен → `Неизвестный`
- Дубликат имени → суффикс `(2)`, `(3)` и т.д.

## Именование процессуальных папок

Формат: `ГГГГ-ММ-ДД Сторона - действие`

Примеры:
- `2026-03-01 Истец - исковое заявление`
- `2026-03-15 Ответчик - отзыв на иск`
- `2026-04-10 к судебному заседанию` (без стороны — общее)

Процессуальные папки — это частный случай папки-комплекта (§«Хронологическая раскладка»): головной документ (исковое, отзыв, ходатайство) + его приложения.

## Папка «Входящие документы/»

Единая точка входа для всех новых файлов в деле. Создаётся при `init-case`.

Правила:
- Сюзерен кладёт сюда любые файлы, которые нужно обработать
- Команды `intake`, `add-evidence`, `add-opponent` берут файлы **только** отсюда
- После успешной обработки файлы из «Входящие документы/» **удаляются** (`rm`); оригиналы при этом уже сохранены в `.vassal/raw/<batch>/`
- Если папка пуста — команда обработки сообщает об этом и завершается

## Удаление обработанных файлов

Claude Code работает в обычной файловой системе, поэтому для удаления используется обычный `rm`:

```bash
rm "Входящие документы/файл.pdf"
```

Оригинал при этом уже скопирован в `.vassal/raw/<batch>/` apply-фазой, так что данные не теряются.

## Хронологическая раскладка «Материалы от клиента/»

Главный принцип раскладки: **одна точка на временной шкале — один элемент в `Материалы от клиента/`.** Тематические папки-рубрики («Договоры», «Претензии», «Переписка», «Акты» и т.п.) — анти-паттерн и запрещены для файлов, у которых есть хронологическая привязка.

### Что именно раскладывается

Единицы раскладки:

1. **Одиночный документ** (нет приложений, не является приложением к другому документу из этого пакета):
   - лежит **файлом** прямо в `Материалы от клиента/`
   - имя строго `ГГГГ-ММ-ДД Отправитель Описание.расш`
   - подпапки не создаются — одиночный документ никогда не оборачивается в папку
2. **Комплект (документ + приложения)** — головной документ, к которому идут приложения (договор + приложения к нему, сопровод + приложенные акты, претензия + приложенные расчёты, исковое + приложения и т.п.):
   - создаётся папка `ГГГГ-ММ-ДД Отправитель Описание/` — имя совпадает с именем головного документа (без расширения)
   - внутри: головной файл (`ГГГГ-ММ-ДД Отправитель Описание.расш`) + приложения
   - приложения именуются `Приложение NN — Краткое описание.расш`, где `NN` — номер приложения из головного документа (`01`, `02`, ...); если номер в исходнике не указан — сквозная нумерация по порядку следования
   - если комплект клиент передал как единую физическую группу (папка/архив) — границы комплекта сохраняются; не разносить по отдельным файлам

### Что не создаёт свою точку на шкале

Приложение, уже входящее в комплект, **не кладётся** отдельным файлом в корень `Материалы от клиента/` даже если у него своя дата: оно лежит только внутри папки головного документа.

### Сироты без даты

Файл — «сирота», если одновременно:

- у него нет читаемой даты (ни в имени, ни в содержимом, ни в метаданных), **и**
- он не входит в чей-то комплект (не является приложением к другому документу этого пакета, не лежал во внутренней физической папке, не распакован из архива-комплекта)

Сироты — единственный случай, когда допустимы тематические подпапки. Раскладка сирот:

- `Материалы от клиента/Без даты — <Тема>/<исходное-или-разумное имя>.расш`
- Тема берётся из содержимого файла: например, `Без даты — Переписка`, `Без даты — Служебное`, `Без даты — Скриншоты WhatsApp`, `Без даты — Архивы без контекста`
- Каждая такая запись в индексе получает флаг `needs_manual_review: true`, чтобы Сюзерен потом вручную расставил даты

Если у файла дата есть, но его не к чему привязать как приложение — это не сирота, а обычный одиночный документ: он ложится файлом в корень `Материалы от клиента/` по дате.

### Скриншоты и изображения — конверсия в PDF

Любой растровый файл (`.png`, `.jpg`, `.jpeg`, `.tif`, `.tiff`, `.bmp`, `.heic`) перед раскладкой **конвертируется в PDF** одностраничный (или многостраничный, если это серия из одного чата/источника).

Правила:
- конверсия выполняется apply-фазой через `python3 [PLUGIN_ROOT]/scripts/image_to_pdf.py --in <путь> --out <путь>`
- в `Материалы от клиента/` попадает только PDF
- оригинальная картинка сохраняется в `.vassal/raw/<batch>/` как есть, рядом с PDF-копией
- имя PDF следует общему правилу: `ГГГГ-ММ-ДД Отправитель Описание.pdf`
- несколько скриншотов одного чата/серии допустимо свести в один многостраничный PDF (это уже комплект из одного источника — раскладывается как одиночный файл по дате первого скриншота)

### Архивы — обязательная распаковка

Любой файл-архив (`.zip`, `.rar`, `.7z`, `.tar`, `.tar.gz`, `.tgz`) **распаковывается** plan-фазой во временную рабочую область `.vassal/work/<batch>/<имя-архива>/`. После распаковки:

- содержимое обрабатывается как обычные «Входящие документы» — OCR, дата, отправитель, классификация
- если в архиве была папка-комплект — её границы становятся границами комплекта в `Материалы от клиента/`
- если в архиве были несвязанные файлы — каждый раскладывается самостоятельно
- сам архив-оригинал копируется в `.vassal/raw/<batch>/` и **не** индексируется как документ (индексируется только его содержимое)
- если архив не распаковывается (пароль, повреждён) — содержимое считается сиротой и идёт в `Материалы от клиента/Без даты — Архивы без контекста/`, плюс `needs_manual_review: true`

Распаковщики: `unzip`, `unrar`, `7z`, `tar` — устанавливаются через `scripts/setup.sh`.

### Что запрещено делать с `Материалы от клиента/`

- Создавать тематические папки (`Договоры/`, `Переписка/`, `Акты/`, `Претензионный порядок/` и подобные) для файлов с привязкой к хронологии. Даже если это «удобнее для навигации» — структуру формирует хронология, а не тематика.
- Дублировать файл в тематическую и хронологическую локацию одновременно.
- Создавать пустые папки «на будущее».
- Переносить в `Материалы от клиента/` содержимое `.vassal/`, `Входящие документы/`.

## Таксономия типов документов

Значения поля `type` в index.yaml:

**Договорные:** договор, дополнительное-соглашение, приложение-к-договору, спецификация, акт, накладная, счёт, счёт-фактура

**Переписка:** претензия, ответ-на-претензию, уведомление, письмо, требование

**Процессуальные:** исковое-заявление, отзыв-на-иск, ходатайство, заявление, возражение, пояснения, дополнение-к-позиции, апелляционная-жалоба, кассационная-жалоба

**Судебные:** определение, решение, постановление, протокол-заседания

**Платёжные:** платёжное-поручение, выписка, справка-из-банка

**Прочие:** доверенность, выписка-из-егрюл, устав, свидетельство, заключение-эксперта, иной-документ

## Источники документов

Значения поля `source` в index.yaml:
- `client` — получено от нашего клиента
- `opponent` — получено от процессуального оппонента
- `court` — получено из суда
- `public` — из публичных источников

## Маршрутизация моделей

Маршрутизация моделей и контракты субагентов описаны в `shared/subagent-dispatch.md`.

## Принцип разделения труда: main-Sonnet vs субагенты

**Main-Sonnet оркестрирует процесс, но не держит в контексте тексты исходников клиента.**

- Main-Sonnet не вставляет содержимое файлов из `Входящие документы/`, `Материалы от клиента/`, `.vassal/raw/` в prompt; вместо этого передаёт субагентам абсолютные пути.
- Main-Sonnet читает метаданные дела, шаблоны и deterministic-output субагентов, а затем сам пишет итоговые записи в `.vassal/*` и пользовательские артефакты.
- Haiku-subagent выполняет механическую рутину и vision-OCR по контракту 3.3, Opus-subagent делает аналитическую работу по контракту 3.1, Sonnet-subagent оформляет `.docx` через `arbitrum-docx` по контракту 3.2.
- В аналитических цепочках субагенты не меняют файлы дела напрямую. Исключения фиксируются в контракте: Sonnet-subagent пишет `.docx` через `arbitrum-docx`, Haiku-subagent может выполнять `rename`, если это явно разрешено main-Sonnet.

## OCR quality thresholds

| Условие | `ocr_quality` |
|---|---|
| `extraction_method ∈ {pdf-text, docx-parse, text-read}` | `ok` |
| `extraction_method == ocr` и `confidence ≥ 0.75` и ≥ 200 символов/стр в среднем | `ok` |
| `extraction_method == ocr` и (`confidence < 0.75` или < 200 симв/стр) | `low` |
| `extraction_method == ocr` и всего < 50 символов | `empty` |
| `extraction_method == haiku-vision` и `confidence ≥ 0.75` и ≥ 200 символов/стр в среднем | `ok` (с `ocr_reattempted: true`) |
| `extraction_method == haiku-vision` и (`confidence < 0.75` или < 200 симв/стр) | `low` (с `ocr_reattempted: true`, `reason: "re-OCR via haiku-vision still low"`) |
| Поле отсутствует (дело `v0.5.x`) | трактуется как `ok` (презумпция) |

Связанные поля в `index.yaml`: `ocr_quality`, `ocr_quality_reason`, `ocr_reattempted`.

## Аудируемость выводов

Каждый юридический вывод агента оформляется:
```
**Вывод:** [заключение]
**Фактическое основание:** [на чём основано]
**Правовое основание:** [нормы, статьи]
**Источники:** [названия документов с датами -- НЕ doc-ID; документ читает человек]
**Уверенность:** высокая | средняя | низкая
**Недостающие данные:** [чего не хватает]
```

## Thinking-override

Main-Sonnet сканирует user-message и форвардит override в THINKING-секцию промпта Opus-субагента:

```text
если в user-message есть "ultrathink"                 → "ultrathink"
иначе если "think harder" / "megathink"               → "think harder"
иначе если "think hard" / "think deeply"              → "think hard"
иначе                                                 → default из таблицы маршрутизации скиллов
```

Дефолты по скиллам:
- `timeline`, `legal-review`, `analyze-hearing`, `draft-judgment`, `appeal`, `cassation` → `think hard`
- `build-position`, `prepare-hearing` → `think harder`
- файловые скиллы и `catalog` → без thinking
