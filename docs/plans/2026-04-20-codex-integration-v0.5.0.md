## Goal

Включить Codex CLI (OpenAI GPT-5 family через OAuth) в качестве полноправного участника пайплайнов vassal-litigator для четырёх ролей: (a) файловый исполнитель с `--write` на medium effort для всей «грязной» работы с документами (intake, update-index, catalog, add-evidence, add-opponent); (b) таймлайн-билдер на high effort; (c) визуализатор через `$imagegen` (gpt-image-1.5) для схем сторон, графов договорных связей, инфографики таймлайна — как **sidecar-превью** (см. Contracts §5, §12); (d) контрольный ревьюер на xhigh в аналитических скиллах (legal-review, build-position, prepare-hearing, draft-judgment, appeal, cassation). Claude-main (Sonnet) остаётся в роли оркестратора: триаж, preview → apply, взаимодействие с Сюзереном, git, мета-контроль. Claude Opus продолжает выполнять правовой анализ, позиции и проекты процессуальных документов.

Визуализации — **жёсткое требование v0.5.0**, не откладываются. Ф5 ДОЛЖНА поставить работающие визуализации любой ценой: либо через `codex-companion.mjs --background` (если $imagegen пробрасывается), либо через прямой `codex exec` без `--background` (теряем удобный Monitor-polling, получаем картинки). Stub-вариант «перенести на 0.5.1» исключён. При этом визуализации — sidecar-only: лежат в `.vassal/visuals/`, НЕ индексируются в `index.yaml`, НЕ встраиваются в процессуальные документы и клиентские отчёты по умолчанию. Они — AI-generated, не доказательство.

Версия 0.5.0 — breaking change. Из маршрутизации моделей полностью убираются Haiku-субагенты на файловую часть (их заменяет Codex medium). ARCHITECTURE §14 п.6 «всё локально, обезличивание не требуется» отменяется: плагин глобально принимает отправку клиентских материалов в OpenAI; вместо обезличивания и opt-in/out — честное предупреждение в README о том, что использование плагина означает согласие юриста на обработку материалов дела в OpenAI. Каждый файл содержит ровно одну ответственность, каждая фаза оставляет плагин установимым и функциональным (хотя часть скиллов может временно работать по старой схеме до подхода соответствующей фазы).

**Заметка по визуализатору (уникальный случай).** Роль imagegen-visualizer — единственная, для которой допустим НЕ-companion путь. Причина: `$imagegen` — это Codex-side runtime-навык (image_gen), и companion.mjs не умеет ни пробрасывать feature-flag `--enable image_generation`, ни выставлять `sandbox workspace-write` для чего-то, кроме `--write`. ## Что ревьюим

```
Путь к аналитическому выходу: [OUTPUT_PATH] (например, `[CASE_ROOT]/Правовое заключение.md` или `[CASE_ROOT]/2026-04-20 Начальная позиция.md`).

## Контекст (читай сам из файлов, не подаю тебе текстом)

- cwd = [CASE_ROOT]. Схемы и конвенции плагина — в `[PLUGIN_ROOT]/shared/` (не в `shared/`).
- `[CASE_ROOT]/.vassal/case.yaml` — карточка дела.
- `[CASE_ROOT]/.vassal/index.yaml` — реестр документов.
- `[CASE_ROOT]/.vassal/analysis/` — все аналитические блоки (если применимо к скиллу).
- Ключевые md-зеркала из `[CASE_ROOT]/.vassal/mirrors/` — выбери сам по тому, на какие документы ссылается анализируемый выход (по названиям в секции «Источники» либо по doc-NNN).
- Исходная задача юриста: [ORIGINAL_INPUT] — то, что Сюзерен передавал в команду.

## Режим ревью (обязательно)

Ты НЕ ищешь code bugs. Ты ищешь **юридические недоработки**:

1. **Пропущенные пресекательные сроки**: проверь, что анализ учитывает общую исковую давность (ст. 196 ГК) и специальные сроки, процессуальные сроки (апелляция 30 дн, кассация 2 мес, предъявление ИД 3 года), течение (ст. 200 ГК), приостановление (ст. 202), перерыв (ст. 203).
2. **Неучтённые нормы**: если анализ ссылается на одну норму, но есть более специальная — отметь. Если пропущена норма о форме / существенных условиях / последствиях несоблюдения — отметь.
3. **Слабые места аргументации**: найдены ли все контраргументы оппонента? Выдержит ли позиция стресс-тест? Есть ли логические разрывы между фактами → нормой → выводом?
4. **Нарушение формата аудируемости**: каждый юридический вывод должен иметь «Вывод → Фактическое основание → Правовое основание → Источники → Уверенность → Недостающие данные» (см. `[PLUGIN_ROOT]/shared/conventions.md`). Найди выводы без этой структуры.
5. **Противоречие материалам дела**: есть ли выводы, которые противоречат тому, что написано в md-зеркалах документов? Конкретные расхождения по датам, суммам, сторонам.
6. **Пропущенные документы**: на какие документы индекса анализ должен был сослаться, но не сослался?
7. **Процедурные ошибки**: правильно ли определена подсудность? досудебный порядок? компетентный суд?

## БЛОКИРУЕТ (BLOCKING) — критерии

- Пропущенный критический срок, который может привести к потере права.
- Применена/не применена норма, искажающая вывод.
- Логическое противоречие в мотивировочной части.
- Ссылка на источник, которого нет в index.yaml (фабрикация).
- Ссылка на статью закона с неверным номером/содержанием.
- Противоречие фактам из md-зеркал.

## НЕ БЛОКИРУЕТ (NITS)

- Стилистика.
- Порядок разделов.
- Отсутствие судебной практики (если Сюзерен не просил).
- «Можно было бы усилить» — это nit.

## Формат вывода (строго, аудируемый)

Первая строка — ровно одно из: `REVIEW_OK` или `REVIEW_BLOCKING`.

Каждое BLOCKING-замечание эмитируй структурированным блоком markdown (ниже — шаблон). Это требование аудируемости ARCHITECTURE §1.2 п.8 — юрист должен понимать, на чём основано замечание и где его проверить.

```
## BLOCKING 1: <краткая тема в 5-7 слов>

- **Категория:** срок | норма | логика | формат | фактическое противоречие | фабрикация | процедура
- **Замечание:** <суть проблемы в 1-2 предложениях — что не так>
- **Фактическое основание:** <какой фрагмент [OUTPUT_PATH] вызывает вопрос. ОБЯЗАТЕЛЬНО цитата (в кавычках) + ссылка на раздел/строку документа>
- **Правовое основание:** <конкретная норма — статья ГК/АПК/ГПК/специального закона с номером пункта/части. Если ни одной нормы не применимо — явно напиши «Нет применимой нормы — логическая/процедурная ошибка»>
- **Источники (doc-NNN):** <перечисли doc-ID из index.yaml, на которые опирается твоё замечание. Если только один — один. Если ни на один — напиши «нет — замечание по внутренней логике документа»>
- **Уверенность:** высокая | средняя | низкая
- **Недостающие данные:** <чего не хватает, чтобы снять замечание. Если ничего — «замечание обоснованно полностью»>
- **Что проверить/уточнить:** <конкретное действие для юриста: открыть такой-то документ, сверить такую-то дату, переформулировать такой-то абзац>
```

После всех BLOCKING-блоков — секция NITS в более лёгком формате (без полного блока, но с указанием места и сути):

```
## NITS

- [место в документе] <краткая суть, 1 строка>
- [место] ...
```

## Дисциплина

- Цитата в «Фактическом основании» — буквальная, не перефраз. Если цитата >3 строк — приводи ключевую подфразу.
- Ссылки на нормы — только реально существующие статьи. **Фабрикация норм** сама по себе делает ревью недействительным — Claude-main откатит.
- «Источники (doc-NNN)» — только ID, которые физически присутствуют в `[CASE_ROOT]/.vassal/index.yaml`. Не придумывай.
- Если всё чисто — `REVIEW_OK` + опционально секция NITS без BLOCKING-блоков.

Быть скрупулёзным на BLOCKING, но не выдумывать.
```

### 7. Формат записи ревью в файловую систему

После каждого ревью xhigh аналитического скилла Claude-main сохраняет отчёт в:

```
[CASE_ROOT]/.vassal/reviews/{ГГГГ-ММ-ДД}-{skill-name}.md
```

(Формат имени — дата впереди для хронологической сортировки.)

Фронтматтер:
```yaml
---
skill: legal-review | build-position | prepare-hearing | draft-judgment | appeal | cassation
target_output: относительный/путь/к/аналитическому/файлу
reviewer: codex-xhigh
reviewed_at: 2026-04-20T15:30:00
verdict: REVIEW_OK | REVIEW_BLOCKING
blocking_count: N
nits_count: M
---
```

Тело — точная копия вывода ревьюера (заголовки `## BLOCKING N: ...` со всеми полями «Замечание / Фактическое основание / Правовое основание / Источники / Уверенность / Недостающие данные / Что проверить», затем `## NITS`). Этот артефакт — юридически аудируемый документ в деле: юрист должен при открытии сразу понимать, что проверять.

### 8. Обработка BLOCKING ревью аналитики (UX-контракт)

При `REVIEW_BLOCKING` Claude-main:
1. Сохраняет отчёт в `.vassal/reviews/`.
2. Показывает Сюзерену верхнюю сводку (количество BLOCKING + их сжатые темы).
3. Предлагает три опции:
   - (a) **Принять как есть**: подтвердить, что юрист согласен игнорировать ревью (агент не может идти против Сюзерена). Файл сохраняется с пометкой `<!-- reviewed: accepted-over-objection -->` в конце.
   - (b) **Один раунд фикса через Opus**: Claude диспатчит Opus с конкретным списком BLOCKING + ссылкой на исходный выход. Opus правит, новый артефакт сохраняется, БЕЗ повторного ревью (цикл исключён).
   - (c) **Править вручную**: Claude показывает BLOCKING-список, Сюзерен правит сам в редакторе, артефакт фиксируется как финальный.

При `REVIEW_OK` — артефакт сохраняется без пометок, NITS показываются Сюзерену как информация.

### 12. Статус визуализаций: sidecar-only, opt-in manual attachment

Визуализации, созданные Codex через `$imagegen`, имеют ЧЁТКИЙ юридический статус: **они AI-generated, это не доказательство и не часть процессуальных/аналитических документов дела**. Следствия для архитектуры:

1. **Место хранения — только `[CASE_ROOT]/.vassal/visuals/`.** Никаких других путей.
2. **Не индексируются.** `.vassal/index.yaml` содержит только документы дела. Картинки sidecar не попадают в индекс. Скиллы и промпты file-executor явно не создают для них записи.
3. **По умолчанию не встраиваются; встраивание — только через явный opt-in юриста.** Default-флоу никакого скилла не добавляет ссылок `![...](.vassal/visuals/...)` в `Правовое заключение.md`, `Хронология дела.md`, любые процессуальные документы, клиентские `.docx`-отчёты. Opt-in для КЛИЕНТСКИХ отчётов — см. §12.5 (редкая ручная процедура с обязательной подписью). Opt-in для ПРОЦЕССУАЛЬНЫХ — запрещён абсолютно (§12.6). Mermaid-блоки в `Хронология дела.md` — допустимы и не относятся к правилу (текстовая диаграмма, воспроизводимая, не AI-generated); правило про встраивание касается только растровых `.png` из `.vassal/visuals/`.
4. **Просмотр юристом — через ручное открытие папки `.vassal/visuals/`.** Это осознанный выбор Сюзерена, не автоматика.
5. **Ручное приобщение в клиентский отчёт** (opt-in, редкий случай) — отдельная процедура:
   - Сюзерен явно запрашивает: «добавь картинку X в отчёт Y».
   - Claude-main показывает предупреждение: «Картинка `{путь}` сгенерирована AI (Codex $imagegen). При приобщении к клиентскому отчёту она должна быть явно помечена как иллюстрация, не как факт или доказательство. Продолжить?».
   - При подтверждении — Claude-main добавляет ссылку на картинку и явную подпись вида `*AI-generated иллюстрация, справочно; не является доказательством в деле*`.
   - Это **отдельный ручной шаг**, НЕ default-флоу никакого скилла.
6. **Процессуальные документы — НИКОГДА.** Даже по явной просьбе Сюзерена в процессуальные документы (`prepare-hearing`, `appeal`, `cassation`, `draft-judgment`) AI-картинки не встраиваются. Это жёсткое правило — процессуальные документы подписываются юристом и идут в суд, там не место AI-graphics с неопределённым статусом.

### 13. Preview → Apply контракт (Variant X — зафиксирован)

Для всех файловых скиллов (intake, update-index, catalog, add-evidence, add-opponent) действует единая модель:

**Preview (Claude-main, read-only):**
- Читает имена файлов в «Входящие документы/» / целевой папке.
- По заголовкам, датам, именам, размерам, типам файлов (extension) — предлагает план: какие файлы куда переносятся, как переименовываются, какая структура папок получится, какие зеркала будут созданы (как заголовки, без содержимого).
- **Без OCR**, без чтения содержимого PDF/JPG/DOCX, без создания md-зеркал.
- **Без** копирования в `.vassal/raw/`.
- **Без** любых файловых операций — ни `cp`, ни `mkdir`, ни записи в `.vassal/index.yaml`, ни вызова `extract_text.py`.
- Результат preview — текст для Сюзерена: «План: переместить X → Y, создать зеркало для Z, обновить индекс так-то».

**Apply (Codex medium, `--write`, после подтверждения Сюзерена):**
- ВСЕ файловые операции, включая копирование оригиналов в `.vassal/raw/{batch}/`, OCR через `[PLUGIN_ROOT]/scripts/extract_text.py`, создание зеркал `.vassal/mirrors/doc-NNN.md`, перемещения, обновление `.vassal/index.yaml` и `.vassal/case.yaml`, запись `history.md`.
- Точно по согласованному плану из [PLAN_BODY].

**Обоснование Variant X (vs. Variant Y):** выбран Variant X — preview без OCR в Claude-main. Причины:
1. Соответствует текущей архитектуре v0.4.0 (preview всегда дешевле, чем apply).
2. Экономит Codex-токены (OCR дорогой; запуск его дважды — раз на preview, раз на apply — удваивает бюджет).
3. Сохраняет жёсткую границу «никаких файловых операций до подтверждения». Даже read-only Codex-вызов с OCR, пишущий результаты в отчёт, создаёт временные файлы в Codex-sandbox и артефакты в логах — их полная read-only гарантия сложнее, чем полное отсутствие Codex в preview.
4. Потеря точности preview (preview «по обложкам») — приемлема: Сюзерен всё равно подтверждает план, и если preview неточен — apply может вернуть `DONE_WITH_CONCERNS` / `NEEDS_CONTEXT`, Сюзерен увидит и скорректирует.

**Жёсткие инварианты (проверяются в smoke):**
- После preview-фазы хэш `.vassal/` и хэш «Входящие документы/» **не изменились**.
- Никаких файлов в `.vassal/raw/` до apply.
- Никаких временных файлов от Codex в папке дела в preview-фазе.

### 14. Путь к helper-скриптам плагина (инвариант)

Единое правило для всех промптов, скиллов, и документации v0.5.0:

- `python3 [PLUGIN_ROOT]/scripts/extract_text.py <путь>` — OCR/извлечение текста.
- `python3 [PLUGIN_ROOT]/scripts/generate_table.py --case-root "[CASE_ROOT]"` — генерация xlsx.
- `bash [PLUGIN_ROOT]/scripts/setup.sh` — установка зависимостей.

**Никогда** не вызывать эти скрипты как `scripts/extract_text.py` без префикса `[PLUGIN_ROOT]/` — из папки дела это относительный путь в пустоту.

Claude-main при формировании промпта подставляет конкретное значение `[PLUGIN_ROOT]` (как text-replace — не shell variable) и передаёт полностью готовый промпт. Codex получает абсолютные пути, без необходимости что-то резолвить сам.

## Test strategy

Плагин vassal-litigator не имеет автоматизированных тестов (pytest/unittest). «Тесты» = smoke-проверки + README-инструкции для ручной верификации.

### Общие инварианты, которые проверяются в smoke-тестах всех фаз

1. `.vassal/case.yaml` и `.vassal/index.yaml` валидны YAML (`python3 -c "import yaml; yaml.safe_load(open(...))"`).
2. `index.yaml` консистентен: все `mirror:` файлы существуют; все `file:` существуют; нет дубликатов ID.
3. Нет файлов с пустыми именами или трансилитерированными русскими.

## Risks / unknowns / assumptions

### R1 (HIGH, частично верифицирован эмпирически в round 3). `$imagegen` и путь сохранения

**Что уже известно (эмпирически, codex-cli 0.121.0, тест 2026-04-20):**

- `$imagegen` = runtime-навык `image_gen` (модель `gpt-image-1.5`). В `codex exec` работает, если включены: (a) feature-flag `image_generation` (через `--enable image_generation` ИЛИ глобально в `~/.codex/config.toml`); (b) `--sandbox workspace-write` (default `read-only` блокирует запись).
- **Путь сохранения:** Codex всегда пишет в `$CODEX_HOME/generated_images/<session_id>/ig_<hash>.png`. `$CODEX_HOME` = `~/.codex` по умолчанию. Пользовательский путь через prompt Codex **игнорирует** для этого навыка.
- Codex возвращает абсолютный путь в своём итоговом stdout — Claude-main парсит его и копирует файл в `[CASE_ROOT]/.vassal/visuals/`.
- Верификация: `codex exec --skip-git-repo-check --sandbox workspace-write --enable image_generation -C /tmp 'Generate ...'` → `/Users/strigov/.codex/generated_images/019dabf0-d749-7f11-92dc-3331fd140781/ig_0314cccc275a54970169e6629904ac81969e428c1ce60b7921.png` (1024x1024 PNG, 72KB, проверено `file`).
- Companion `task --background --write` в принципе может пробросить `$imagegen` как часть prompt (т.к. он просто передаёт строку в `codex exec`), НО companion.mjs v1.0.3 НЕ умеет пробрасывать `--enable image_generation` (проверено в исходнике `scripts/codex-companion.mjs` — `image_gen`/`feature`-ключи не упоминаются). Поэтому Branch A требует, чтобы юрист однократно включил feature-flag глобально.

**Неизвестно (остаётся для Ф0):**

- Пробросит ли companion `$imagegen` в реальном running-прогоне (теоретически должен — но empirical тест в рамках Ф0 обязателен).
- Готов ли юрист включить `features.image_generation = true` глобально в своём `~/.codex/config.toml` (конфигурационный выбор).

**Исходы и реакция плана:**

- **(a) Branch A:** companion пробрасывает `$imagegen` + юрист согласен включить feature-flag глобально → используем единый companion-путь везде.
- **(b) Branch B:** companion не пробрасывает ИЛИ юрист не хочет включать feature-flag → прямой `codex exec --enable image_generation ...` (команда полностью верифицирована, см. выше). Цена — синхронный блок Claude-main на ~30-90 сек, нет Monitor poll.
- **(c) Вообще не работает на нашей конфигурации** — крайне маловероятно (Branch B уже работает эмпирически). Stub-вариант / перенос на 0.5.1 исключён по требованию автора. Если Ф0 попал в исход (c) на конкретной машине юриста — это блокер релиза ДЛЯ ТОЙ МАШИНЫ (не архитектурная проблема плагина): нужна диагностика среды юриста (версия codex-cli, доступность OAuth, квоты).

**Mitigation:** Ф0 обязательно заканчивает `docs/research-report.md` с чётким ответом по трём подвопросам (см. §Ф0.1: 1a/1b/1c) и выбирает Branch A или B. Ф5 стартует только после ответа.

### R11 (HIGH — юридический риск, не технический). Визуализация как «доказательство»

Риск: Codex-generated картинки (схема сторон, граф договорных связей, инфографика таймлайна), попав в процессуальный документ или клиентский отчёт, могут быть восприняты адресатом (судом, клиентом) как часть доказательной базы / достоверных материалов дела. Это недопустимо: картинки — AI-generated, не факт, не документ, не выдерживают проверку на аудируемость.

**Mitigation (реализовано в архитектуре плана, не добавляется отдельно):**
- Sidecar-only хранение в `.vassal/visuals/` (Contracts §12.1).
- НЕ индексируются в `index.yaml` (§12.2) — не могут быть случайно упомянуты в процессуальных документах через механизмы «список документов».
- НЕ встраиваются в `Правовое заключение.md`, `Хронология дела.md`, процессуальные документы, клиентские отчёты по умолчанию (§12.3, §12.6).
- Процессуальные документы — абсолютный запрет, даже по явной просьбе (§12.6).
- Opt-in ручное приобщение в клиентский отчёт — с явной подписью `*AI-generated иллюстрация, справочно; не является доказательством в деле*` (§12.5).
- Смысловой сигнал Сюзерену: при каждом предложении визуализации в preview — текст «не является доказательством».
- Smoke-тест Ф5: ни один аналитический артефакт не содержит ссылок на `.vassal/visuals/*` после прогона с включёнными визуализациями.

**Assumption:** Сюзерен — юрист и не будет сознательно приобщать AI-картинку как доказательство. Политика плагина: не давать делать это случайно/механически.

### R2 (MEDIUM). Стабильность Codex medium в YAML-мутациях `.vassal/index.yaml`

Риск: Codex при большом объёме intake (50+ документов) может сломать YAML (неправильный отступ, дублирующиеся ключи, утерянные поля). Последствия — весь индекс нечитаем, дело зависает.

**Mitigation:**
- В каждом file-executor промпте **обязательна** финальная YAML-валидация (`yaml.safe_load`).
- Политика: если валидация после mutation упала — Codex **откатывает** изменение (из `.vassal/index.yaml.bak`, который создаёт ПЕРЕД каждой записью). Claude-main проверяет, что отчёт содержит успешную валидацию.
- Assumption: Codex умеет создавать и откатывать .bak-файлы. Это тестируется в Ф2 на fixture с намеренной кривой вставкой (тест `tests/smoke/test-yaml-resilience.sh`, необязательный но желательный).
- Если проблема воспроизводится часто — fallback: Codex делает только структурные изменения (new docs, deletes), а мерж выполняет Python-скрипт `scripts/merge_index.py` (новый, не в скоупе v0.5.0 — в 0.5.1, если понадобится).

### R3 (MEDIUM). Сохранение русских имён папок/файлов при операциях Codex

Риск: Codex транслитерирует «Входящие документы/» в "Incoming documents/" или ломает UTF-8. Последствия — плагин перестаёт работать с реальными делами.

**Mitigation:**
- В промптах явный запрет: «НЕ трансилитеруй, работай в UTF-8, цитируй пути». Это первый пункт дисциплины.
- Ф0 проверяет на fixture с русскими именами, что Codex medium не ломает имена.
- Smoke-тест Ф2 включает явную проверку: после intake — папка «Материалы от клиента/Договоры/» существует с кириллицей (не «Materialy ot klienta/»).

### R4 (LOW-MEDIUM). Hardcoded путь к версии companion (1.0.3)

Риск: openai-codex плагин обновится (1.0.4, 2.0.0 и т.д.), наш путь сломается. Сюзерен не сразу поймёт, почему отвалились все Codex-скиллы.

**Mitigation:**
- В `codex-invocation/SKILL.md` путь вынесен в **один** блок с явной оговоркой «При обновлении openai-codex замени версию в этой строке».
- Версия вынесена как отдельная markdown-секция «Версия companion», чтобы можно было grep'ать.
- В `README.md` предпосылка «Плагин openai-codex ≥ 1.0.3» плюс заметка «Если после обновления codex плагин перестал работать — см. `skills/codex-invocation/SKILL.md` #hardcoded-path».
- Assumption: мы не пытаемся автоматически резолвить путь по glob. Автоматический резолвер — over-engineering для 0.5.0. В 0.5.1 можно добавить.

### R5 (MEDIUM). Длина промптов для Codex CLI

Codex CLI должен принимать многотысячные промпты (ARCHITECTURE.md имеет ~1000 строк, некоторые скилльные промпты + контекст дела могут давать 20-30k токенов). Неизвестно, есть ли hard-limit.

**Mitigation:**
- Ф0 проверяет: подаём в Codex medium промпт на 30k токенов (реальный ARCHITECTURE + case.yaml + фрагменты зеркал). Фиксируем, прошёл или нет.
- Если не прошёл — в промптах Codex НЕ передаём ARCHITECTURE целиком. Вместо этого Codex читает `[PLUGIN_ROOT]/shared/conventions.md` и `[PLUGIN_ROOT]/shared/*-schema.yaml` САМ с диска — это всегда локальные файлы плагина.
- Контракт промпта (Contracts §9 и `_preamble.md`): Codex читает из plugin root по указке, а не из промпта. Это и основа R5 mitigation, и инвариант всех ролей.

### R6 (MEDIUM). Лимиты Codex при massive-batch intake (100+ документов за раз)

Риск: 100+ документов за один intake могут превысить окно контекста Codex medium (все зеркала не поместятся).

**Mitigation:**
- Claude-main перед диспатчем **батчирует**: если файлов > 30, делит на пакеты по 25-30 и диспатчит Codex несколько раз с `--resume-last` между батчами.
- Либо: Codex НЕ загружает зеркала в контекст, а работает только с именами/метаданными, читая зеркала only-on-demand через `extract_text.py`. Это решение по умолчанию.
- Assumption: на реальных делах юриста редко бывает 100+ документов в один intake, обычно 5-30.

### R7 (LOW). Конфликт с Haiku-субагентами в legacy-скиллах

Риск: до Ф3 часть скиллов (catalog, add-evidence, add-opponent) работает на старой логике с Haiku, а intake/update-index (Ф2) — уже на Codex. Пользователь может получить смешанное поведение.

**Mitigation:**

### Архитектурные развилки, решённые в этом плане (обоснования — здесь)

**A1. codex-invocation — отдельный скилл, не shared/.**
Мотив: (а) скилл автоматически подтягивается Claude через description, если он нужен, — нам нужна именно такая механика для первого вызова Codex в сессии; (б) симметрично с superpowers (`skills/codex-invocation/`); (в) `shared/` по соглашению проекта содержит схемы и шаблоны данных (`case-schema.yaml`, `mirror-template.md`), а не инструкции для агента. Контракт + инструкция — это не данные.

**A2. `prompts/` на уровне плагина, не внутри `skills/codex-invocation/`.**
Мотив: промптами пользуется много скиллов (intake, update-index, catalog, add-evidence, add-opponent, timeline, visualize, все шесть аналитических). Если положить внутрь `codex-invocation/prompts/` — это запутает: codex-invocation становится «скиллом всего Codex», что нарушает принцип одной ответственности. Положили на уровень плагина: `plugins/vassal-litigator/prompts/`, симметрично `commands/` и `skills/`.

**A3. Зависимость openai-codex — README prerequisite, не манифест.**
Решение: в Ф0 проверяем, поддерживает ли Cowork plugin format секцию `dependencies:` в `plugin.json`. Если да — добавляем. Если нет (это вероятнее) — только README. Не тратим время на схемы Cowork вне Ф0.

**A4. Отчёт Codex→Claude для файлового executor — адаптированный.**
Мотив: базовый контракт `DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT` из superpowers идеально подходит, но нужны специфичные поля (FILES_CREATED, FILES_MOVED, INDEX_CHANGES и т.д.). Это зафиксировано в §3 Contracts.

**A5. Минимальный контекст для xhigh-ревью аналитики — **по ссылке**, не полным payload.**
Мотив: экономия токенов и времени. Codex xhigh работает в той же файловой системе — он САМ читает `case.yaml`, `index.yaml`, `.vassal/analysis/`, ключевые зеркала. Claude-main передаёт только ПУТЬ к целевому артефакту + исходный input юриста. Это стабильно: при росте дела контекст не раздувается.

**A6. BLOCKING в ревью аналитики — юридические, не технические.**
Критерии BLOCKING для юридического ревью формализованы в `prompts/analytical-reviewer.md` (§6 Contracts). Ключевое: это не «нашёл баг», это «пропущен срок, искажён вывод, фабрикован источник».

**A7. Один раунд ревью, без цикла Opus↔Codex.**
Мотив: юрист явно написал «только не несколько ревью, долго». Цикл исключён. При BLOCKING — 3 опции для Сюзерена, и на этом всё.

**A8. Ревью только после аналитических скиллов, не файловых.**
Мотив: файловые скиллы и так под preview Сюзерена. Ревьюить переименование файла — over-engineering. Ревью нужно там, где генерируется юридическое содержимое.

## Ф5: Визуализации через $imagegen

**Цель:** добавить скилл visualize и интегрировать в 3 аналитических скилла (legal-review, build-position, timeline). Визуализации ДОЛЖНЫ работать в v0.5.0 — это жёсткий scope. Stub/перенос на 0.5.1 не допускается.

**Файлы:** см. §Files/Ф5.

**Процедура.** Финальный выбор техники вызова зависит от Ф0/R1 — но ОБА возможных исхода ведут к рабочим визуализациям:

**Branch A — `$imagegen` работает через `codex-companion.mjs --background`:**

- Используется единый контракт вызова Codex (companion + Monitor poll).
- `skills/codex-invocation/SKILL.md` для $imagegen не получает исключений — стандартный путь.

**Branch B — `$imagegen` работает только через прямой `codex exec`:**

- `skills/codex-invocation/SKILL.md` содержит **задокументированное исключение** для роли `imagegen-visualizer`: синхронный `codex exec` без companion. Обоснование — R1 Ф0 (прямая ссылка на research-report).
- Следствие: теряется удобный Monitor-polling, Claude-main блокируется на время рендера (~30-90 сек эмпирически). Это приемлемая плата за получение картинок.
- `skills/visualize/SKILL.md` содержит **Bash-вызов прямой**, не через companion. Верифицированный формат команды (эмпирически на codex-cli 0.121.0):

  ```bash
  codex exec \
    --skip-git-repo-check \
    --sandbox workspace-write \
    --enable image_generation \
    -C "[CASE_ROOT]" \
    -c model_reasoning_effort=medium \
    '<полный промпт с $imagegen ..., БЕЗ указания output path>'
  ```

  **Пояснение флагов (все проверены через `codex exec --help` и empirical runs):**
  - `--skip-git-repo-check` — **обязателен.** `[CASE_ROOT]` — обычная папка дела на машине юриста, НЕ git-репо. Без флага `codex exec` откажется запускаться.
  - `--sandbox workspace-write` — обязателен, т.к. `$imagegen` требует прав записи (в `$CODEX_HOME/generated_images/` — не в `[CASE_ROOT]`, но runtime проверяет общий sandbox-режим). Default `read-only` блокирует генерацию.
  - `--enable image_generation` — активирует feature-flag на время этого вызова (эквивалент `-c features.image_generation=true`). Без него `$imagegen` в prompt будет проигнорирован.
  - `-C "[CASE_ROOT]"` — рабочая директория Codex. Claude-main подставляет абсолютный путь. (В `codex exec` это именно `-C/--cd`, как видно в `codex exec --help`.)
  - `-c model_reasoning_effort=medium` — effort для визуализатора (default в user config может быть `xhigh` — его нужно явно понизить, чтобы не тратить квоту).
  - Prompt — последний позиционный аргумент. БЕЗ указания output path (Codex всё равно его проигнорирует — см. §5 Contracts).

- **Два этапа после возврата Codex-команды** (Claude-main ответственность):
  1. Парсит строку `GENERATED_IMAGE: <абсолютный путь>` из stdout Codex.
  2. Копирует файл в `[CASE_ROOT]/.vassal/visuals/{ГГГГ-ММ-ДД}-{visual_type}.png` (см. §5 Contracts пункт «Процедура Claude-main после получения отчёта»).
