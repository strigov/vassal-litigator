# analyze-hearing — Анализ транскрипции заседания

Скилл оркестрируется **main-Sonnet**. Нормализацию входа, сохранение файлов и обновление метаданных делает main-Sonnet; глубокий юридический анализ делает **Opus-subagent** по контрактам 3.0 и 3.1c из [shared/subagent-dispatch.md](/Users/strigov/Documents/Claude/Projects/Suzerain/plugins/vassal-litigator/shared/subagent-dispatch.md).

Ключевой принцип: транскрипция сохраняется как отдельный артефакт заседания и анализируется как самостоятельный источник. В этой ветке нет `.docx`, нет Sonnet-subagent и нет отдельного «клиентского brief» через formatter.

## Предусловия

- Существует `.vassal/case.yaml`.
- Доступна текстовая транскрипция: файл, текст из чата или документ во `Входящие документы/`.
- Доступен `Task(model=opus)` по контракту 3.1c.

## Переменные сессии

Определи в начале:

- `hearing_date` = дата заседания, если она надёжно определяется из транскрипции или аргументов; иначе текущая дата `YYYY-MM-DD`
- `hearing_dir` = `<case_root>/hearings/{{hearing_date}}`
- `transcript_path` = `{{hearing_dir}}/transcript.md`
- `analysis_path` = `{{hearing_dir}}/analysis.md`

## Фаза 1 — Сбор и нормализация (main-Sonnet)

1. Определи источник транскрипции:
   - путь к файлу в аргументах;
   - текст, вставленный прямо в чат;
   - файл во `Входящие документы/`, если юрист явно указывает на него.
2. Прочитай `.vassal/case.yaml` и `.vassal/index.yaml` для контекста дела.
3. Сохрани транскрипцию в `{{transcript_path}}`:
   - если это текст из чата — запиши его в markdown без потери абзацев;
   - если это текстовый файл — скопируй содержимое как markdown;
   - если это markdown-файл — сохрани как есть.
4. Если вместе с транскрипцией передан официальный протокол или иные материалы по заседанию, сохрани их рядом в `{{hearing_dir}}/`.

## Фаза 2 — Preview (main-Sonnet)

1. Быстро оцени материал без глубокого анализа:
   - определилась ли дата заседания;
   - есть ли имена участников;
   - есть ли явные процессуальные решения;
   - есть ли сомнительные фрагменты, которые нужно помечать `[?]`.
2. Покажи Сюзерену preview:
   - куда будет сохранён анализ: `{{analysis_path}}`;
   - какие источники попадут в контекст Opus-subagent;
   - какие подтверждённые изменения могут попасть в `.vassal/case.yaml` после анализа:
     - дата следующего заседания;
     - новые события в timeline с пометкой `source: transcript`;
     - упомянутые приобщения/процессуальные решения.
3. Дождись подтверждения. Если пользователь просит только быстрый устный разбор без записи файлов, не применяй этот скилл: это вне контракта Ф4.

## Фаза 3 — Apply (main-Sonnet + Opus-subagent)

1. Подготовь prompt Opus-subagent строго по контрактам 3.0 и 3.1c:
   - `ROLE`: `Ты юрист-аналитик в скилле analyze-hearing.`
   - `THINKING`: `think hard` по умолчанию или override из [shared/conventions.md](/Users/strigov/Documents/Claude/Projects/Suzerain/plugins/vassal-litigator/shared/conventions.md).
   - `CONTEXT`:
     - `case_root`;
     - абсолютные пути к `.vassal/case.yaml`, `.vassal/index.yaml`, `.vassal/mirrors/doc-*.md`;
     - абсолютные пути к файлам внутри `{{hearing_dir}}/**`, включая `transcript.md` и при наличии протокол.
   - `TASK`: проанализировать ход заседания, речевые паттерны участников, тактику оппонента и реакции суда; сомнительные места из плохой транскрипции помечать `[?]`; опираться только на указанные файлы; не предлагать `.docx`.
   - `OUTPUT`: единый markdown без строки `READY_FOR_DOCX`.
2. Рекомендуй Opus-subagent следующую структуру верхнего уровня:
   - `## Ход заседания`
   - `## Речевые паттерны`
   - `## Тактика оппонента`
   - `## Реакции суда`
   - `## Выводы`
   - `## Следующие шаги`
3. Вызови `Task` с параметрами:
   - `subagent_type: "general-purpose"`
   - `model: "opus"`
   - `description`: 3-5 слов
4. Проверь OUTPUT:
   - markdown не пустой;
   - есть верхнеуровневая секция `## Ход заседания`;
   - есть минимум ещё одна верхнеуровневая предметная секция из рекомендованного набора: `## Речевые паттерны`, `## Тактика оппонента`, `## Реакции суда`, `## Выводы`, `## Следующие шаги`;
   - нет `READY_FOR_DOCX`;
   - OUTPUT не является служебным сообщением об ошибке или извинением: явно отвергай ответы, которые начинаются или в основном состоят из фраз вроде `Извините`, `Не могу`, `Не удалось`, `Ошибка`, `I’m sorry`, `Error`.
5. Если хотя бы одна проверка не проходит, перезапусти Opus-subagent один раз с напоминанием о контракте 3.1c и обязательных секциях. Если повторно не удалось — остановись и покажи блокер.
6. Сохрани markdown как есть в `{{analysis_path}}`.
7. На основании анализа предложи и, после подтверждения Сюзерена, внеси обновления в `.vassal/case.yaml`:
   - `next_hearing`, если дата надёжно извлечена;
   - новые элементы `timeline` с `source: transcript`;
   - статус дела или иные подтверждённые процессуальные изменения.
8. Добавь запись в `.vassal/history.md` о разборе заседания.

## Фаза 4 — Verify (main-Sonnet)

1. Проверь, что существуют:
   - `{{transcript_path}}`;
   - `{{analysis_path}}`.
2. Прочитай `{{analysis_path}}` и проверь:
   - файл не пустой;
   - есть `## Ход заседания`;
   - есть минимум ещё одна предметная секция верхнего уровня;
   - нет строки `READY_FOR_DOCX`;
   - файл не выглядит как служебное сообщение об ошибке или извинение.
3. Если менялся `.vassal/case.yaml`, проверь его YAML-валидность и ограничь изменения только подтверждёнными фактами.
4. Покажи Сюзерену короткое резюме:
   - путь к `{{analysis_path}}`;
   - 3-5 главных выводов;
   - какие поля дела были обновлены.

## Дисциплина

- Не создавать `.docx`, не предлагать formatter и не диспатчить Sonnet-subagent по контракту 3.2.
- Не плодить отдельные `internal.md`, `client-brief.md`, `judge-profile.md`, `opponent-analysis.md`: в Ф4 итоговый аналитический артефакт ровно один — `hearings/YYYY-MM-DD/analysis.md`.
- Не вставлять тексты транскрипции и зеркал inline в prompt main-Sonnet; передавай только абсолютные пути.
- Не обновлять `.vassal/case.yaml` без подтверждения Сюзерена на спорных данных.

## Блокер при недоступном Opus-subagent

Если `Task(model=opus)` недоступен, завершился ошибкой или дважды вернул пустой/невалидный markdown:

- сообщи Сюзерену: `Opus-subagent недоступен, analyze-hearing остановлен. Нужен рабочий Task(model=opus) по контракту 3.1c.`
- покажи конкретную ошибку;
- не переключайся на `.docx`-ветку и не замещай Opus аналитикой main-Sonnet.
